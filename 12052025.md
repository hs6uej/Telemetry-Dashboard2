## 1. Backend — NestJS API

### ทำอะไรไป

สร้าง REST API ด้วย **NestJS + TypeScript** ประกอบด้วย 4 module หลัก

```
backend/src/
├── main.ts              ← จุดเริ่มต้น server + Swagger config
├── app.module.ts        ← root module รวม TypeORM
├── entities/            ← TypeORM database entities
│   ├── user.entity.ts
│   ├── station.entity.ts
│   ├── station-reading.entity.ts
│   └── activity-log.entity.ts
├── auth/                ← JWT authentication
├── stations/            ← จัดการสถานีวัดน้ำ
├── readings/            ← ข้อมูล time-series (TimescaleDB)
└── users/               ← จัดการผู้ใช้ (admin only)
```

### ทำไมถึงเลือก NestJS

- **Swagger built-in** — `@nestjs/swagger` generate docs อัตโนมัติจาก decorator ไม่ต้องเขียน YAML เอง
- **TypeScript native** — type-safe ตั้งแต่ entity ถึง HTTP response
- **Modular** — แยก auth / stations / users / readings ชัดเจน

### Auth Module — JWT

**ทำอะไร:** แทนที่ session-based auth เดิมด้วย JWT Bearer Token

**เหตุผล:** ตาม requirement ที่บอกว่า "ใช้ user token คลุมทุก function" และเพื่อให้ตรงกับรูปแบบ REST API มาตรฐานที่ client ต้องการ

**ไฟล์สำคัญ:**
- `auth.service.ts` — login, register, getMe
- `jwt.strategy.ts` — validate token ทุก request
- `jwt-auth.guard.ts` — guard สำหรับ protect routes
- `roles.guard.ts` — guard สำหรับตรวจสิทธิ์ admin

**Flow การ auth:**
```
1. POST /api/auth/login  →  รับ { access_token }
2. ทุก request ต่อไป    →  Header: Authorization: Bearer <token>
3. Server validate token  →  อนุมัติหรือ 401
```

**Endpoints:**
```
POST /api/auth/login      ← เข้าสู่ระบบ รับ JWT
POST /api/auth/register   ← สมัครสมาชิก (รอ admin อนุมัติ)
GET  /api/auth/me         ← ดูข้อมูลตัวเอง [JWT]
```

### Readings Module — TimescaleDB

**ทำอะไร:** บันทึกและดึงข้อมูล sensor แบบ time-series

**เหตุผล:** ข้อมูลระดับน้ำ/ฝนถูก record ทุกๆ รอบ (เช่น ทุก 15 นาที) มีปริมาณมาก TimescaleDB เก็บ time-series ได้เร็วกว่า PostgreSQL ปกติมาก และมีฟีเจอร์ aggregate ในตัว

**Endpoints:**
```
GET  /api/readings/latest              ← ค่าล่าสุดทุกสถานี [JWT]
GET  /api/stations/:id/readings        ← ประวัติค่าสถานี (default 24h) [JWT]
GET  /api/stations/:id/readings?hours=168 ← ย้อนหลัง 7 วัน [JWT]
POST /api/stations/:id/readings        ← บันทึกค่าใหม่ [JWT]
```

**ตัวอย่าง POST บันทึกค่า:**
```json
POST /api/stations/1/readings
Authorization: Bearer <token>
{
  "waterLevel": 6.85,
  "rainLevel": 12.4,
  "flowRate": 145.2,
  "time": "2025-05-12T14:00:00Z"   ← optional (default = now)
}
```

## 2. Database — TimescaleDB

### ทำอะไรไป

เปลี่ยนจาก PostgreSQL ธรรมดาเป็น **TimescaleDB** และออกแบบ schema ใหม่

```
db/init.sql — สร้างทุกตารางพร้อม seed data
```

### Tables

| ตาราง | ประเภท | ใช้ทำอะไร |
|-------|--------|-----------|
| `users` | ปกติ | ผู้ใช้ระบบ |
| `stations` | ปกติ | สถานีวัดน้ำ |
| `station_readings` | **Hypertable** | ค่า sensor แบบ time-series |
| `activity_logs` | ปกติ | log การใช้งาน |

### TimescaleDB Hypertable

```sql
-- station_readings เป็น hypertable
-- แบ่ง partition ตามเวลาอัตโนมัติ
SELECT create_hypertable('station_readings', 'time');

-- Continuous aggregate: hourly average
-- คำนวณค่าเฉลี่ยรายชั่วโมงอัตโนมัติ
CREATE MATERIALIZED VIEW station_readings_hourly ...
```

**ทำไมถึงสำคัญ:** สถานีวัดน้ำ 100 แห่ง บันทึกทุก 15 นาที = 400 records/ชั่วโมง = 3.5 ล้าน records/เดือน TimescaleDB จัดการได้เร็วกว่า Postgres ปกติหลาย 10 เท่า

### lib/api.ts — API Client

**ทำอะไร:** wrapper สำหรับเรียก backend API ทุก request

**ทำไม:** ไม่ต้องเขียน fetch + header + error handling ซ้ำในทุก component

**ตัวอย่างการใช้:**
```typescript
// ดูสถานีทั้งหมด
const stations = await api.stations.list()

// บันทึกค่าใหม่
await api.readings.add(stationId, {
  waterLevel: 6.85,
  rainLevel: 12.4
})

// ดูประวัติ 7 วัน
const history = await api.readings.get(stationId, 168)
```

**Error handling อัตโนมัติ:**
```
ถ้า response = 401 → ลบ token + redirect /login
ถ้า response error → throw Error พร้อม message
```

## 4. Docker Compose

### ทำอะไรไป

สร้าง `docker-compose.yml` รวม 3 services

```yaml

  db:       timescale/timescaledb:latest-pg16  → port 15432
```

**Health checks:**
- db → pg_isready ก่อน backend เริ่ม
- backend → /api/docs-json ก่อน frontend เริ่ม

**ทำไม:** ตาม requirement "deploy ลง docker ทั้งหมด" — ทุก service อยู่ใน docker network เดียวกัน
